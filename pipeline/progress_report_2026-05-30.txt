PROGRESS TIMELINE TODAY
-----------------------

Start of day: 362 foundations (overcounting)
  Root cause: Vision AI running on BOTH overview page (1:250) AND detail pages
  (1:100) which are zoomed sections of the same area. All stacked as separate
  buildings → 2.3× overcounting.

  Fix: Added _assign_foundation_roles() — detects OVERALL vs DETAIL pages by
  scale ratio (max >= 1.5x min). DETAIL pages skip Vision AI, OVERALL pages
  run it. Commit: dbfa74d

After fix 1: 213 foundations (still overcounting)
  Root cause: Vision AI was reading the schedule TABLE (a list of foundation
  types with dimensions in the corner of the drawing) as placed foundations.
  E.g., listing "PG1, PG2, L-D1, L-D2..." as plan annotations.

  Fix: Added CRITICAL DISTINCTION section to prompt. Clearly separated:
  TYPE 1 (plan annotations — COUNT THESE) vs
  TYPE 2 (schedule table entries — IGNORE COMPLETELY)
  Also added schedule-guided filter: if schedule has ≥3 known marks, drop
  any Vision AI result whose label isn't in the schedule.

After fix 2: 122 foundations (undercounting)
  Root cause 1: Gemini outputs Python literals (None, NaN, True) inside JSON
  → json.loads fails → partial recovery stops at object 17 of ~50.
  Fix: Added regex normalisation in _parse_json(): None→null, NaN→0, etc.

  Root cause 2: Y-axis grid was never populated from Vision AI because the
  condition was "if NOT x AND NOT y" — when X existed, Y was never fetched.
  Fix: Changed condition to "if NOT y" so Vision AI always provides Y-axis.

  Root cause 3: Download button disappears after clicking (Streamlit rerun
  clears state). Fix: Save job results to session_state, render from there.

After fix 3: 122 foundations (still undercounting)
  Root cause: Gemini's internal output limit is ~2600 chars. With 2x2 tiles,
  each tile had ~50 foundations × 120 chars/object = 6000 chars → truncated
  after object 17. Context logging confirmed: Gemini closed the JSON array
  mid-object with `"grid_ref":]` at exactly char 2599.

  Fix: Switched to 3x3 tiling (9 tiles instead of 4). Each tile now has
  ~17 foundations × 40 chars = 680 chars — well under the 2600 limit.
  Also compacted the prompt schema to drop verbose fields.

After fix 4: 156 foundations ✓ (target: 157, delta = 1)
  COUNT is now essentially correct.
  However: POSITIONS are still scattered/wrong in SketchUp.

  Root cause (identified, fix in progress): The compact schema dropped
  x_percent/y_percent for on-grid foundations. When grid_ref label matching
  fails (Gemini returns "10/11" but grid has different label format), code
  falls back to the tile CENTER coordinates (0.5) instead of actual position.

  Fix applied (commit pending): Restored x_percent/y_percent for ALL objects
  in the compact schema. 17 objects × 70 chars = 1190 chars — still safe.
  When grid snap works → exact position. When it fails → Gemini's x/y %.
